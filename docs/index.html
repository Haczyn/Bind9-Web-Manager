General Informations

I hope you will get a bit of understanding how this process is to be set up with this description. The software is here to make replication with DNS simpler and more transparent and controlable. I made the process to achieve this, as simple as it was possible for me and I hope with my instructions you are able to achieve the goal you are trying to reach with my dns tool, if you have in mind using it. Below Described are some Features and configuration / use of that Software. If may have a tutorial video on may channel, check out my channels playlist described at the end of the documentation to know more! (I am not sure if i will make one, but if i will do you will find the tutorial video at the end of the documentation with an url to the playlist.)

Slave server replication

With this Panel you can Control Replication between Slave and Master Servers, if there is bind9 running on both machines. More informations how to setup Connections below. You will have a detailed overview over conflicts, insights and control about the replication! You can always see the status of the last replication process in the "Replications" Tab of the website! At the Servers page you have different Buttons to check if a Server is responding and security keys are working.

Slave/DNS Server Hybrid Support

One Server can be Slave and Master Server at the same time. If you register a Master Server in the Servers Section. The Server where you registered that, will act as a Slave Server and replicate Domains from that Master Server.

User Management

This software comes with usermanagement in it. You can create/edit/delete users and assign different permissions of different areas to them.

Rights Management

User Rights management to jail users in theyre operational area, if needed!

IP Blacklisting

You can control IP Blacklisting in the "Blocklist" Area of the webpage. IP Counters get raised for fail logins, fail api token requests. If you want to unblock an ip you can do that in the panel, our automate the process by running the daily.php cronjob daily. This will reset this table on a daily base (optional). If an ip has reached its limit it can not make API Requests anymore and not login to this panel. (No Replication Possible if Server IP is blocked)

Non-Destructive

This software works non-destructive! It does not change the main files, it will create own configuration files for functioning. If something on your system goes wrong with this panel, there always should be a way to rollback! Configuration Files which will may be overwritten by this panel will be secures at the same location they were under a different name to prevent issues. You only have to change one bind file by yourself and we are all good! If you want to rollback, you just need to restore your named.conf file in /etc/bind. Than no changes made by this panel will be in affect anymore. This can save a broken configuration fast if your dns system has errors related to this software (but only on master servers, if you change configuration on slave servers the slave domains wont be registered anymore)

Installation

Here are informations on how to install this panel.

Requirements

  • Tested on Ubuntu and Debian (may work on other systems, but untested)
  • Root Permissions for Cronjob on System
  • Apache2 Webserver with PHP 7/8 Support
  • Apach2 Modules: rewrite, headers
  • PHP Modules: mysql, curl, intl, mbstring, zip, gd
  • Access to MySQL Database (with full permission for that database)
  • Open Ports 53/953 (TCP/UDP) [DNS]
  • Open Port 80/442 (TCP) [WEB]

Deployment Steps

  • Upload the files in the "source" directory of this repository to your webspace.
  • Check Settings.sample.php and provide valid MySQL User Informations, the current Nameservers Hostname and others (see comments).
  • Check Settings.sample.php and read the Notes and comments to understand what the different settings do, this is very urgent, as this settings will control how the dns service will act and which files will be used.
  • Rename settings.sample.php to settings.php
  • Login with Default Credentials (see below in this readme)
  • Set up the Cronjobs as described below.
  • Change your /etc/bind/named.conf to look like this (remove everything else!) :

    include "/etc/bind/dnshttp/dnshttp.named.conf";

    Make a backup of the old file, if you want to deactivate all functionality of this panel for bind, just rollback to the old file. If you have changed this related file in the code setup you need to ajust this setting. But this setting fits for a default configuration and installation. It is not recommended to change this.
  • You are now ready to use all functionalitys of this software! No need to setup MySQL Tables by yourself, the software will do that without installation! You can login now at the domain your website is deployed on...

Setup Cronjobs

Please set up both cronjobs as root user!
Command Interval Description
PATH_TO_WEBROOT/_cronjon/daily.php >/dev/null 2>&1; daily Resets Blacklisted IPs for Webinterface and API Requests (Optional but Recommended)
PATH_TO_WEBROOT/_cronjon/sync.php >/dev/null 2>&1; hourly Needed for DNS Server Configuration Operations and Database Synchronisation + Replication + Autosetup
This cronjob will restart bind and set permissions to folders and zone files when ready... You can set this cronjob to run in an interaval you wish. One Hour Interval is recommended for 1 hour fast replication.

Adjust Bind Configuration named.conf

Keep in mind to edit your /etc/bind/named.conf file to make this panel work. Make a backup of this file before to rollback if anything goes wrong with this dns panel. Or to deactivate it in the future if you dont need it anymore.

Informations About Domains and Replications

Domains, Replications and Conflicts

You can see replicated Domains in the Domains->Slave Domains Section. There are all domains which have been copied from other dns servers, which are registered in the Replication->Manage DNS Servers Area! The data in the interface you see may outdated. It updates every time the cronjob dies execute, you will see the last time of execution on top of the domain pages. There are a lot of icons which describe the status of servers or domains, you can see what the icons mean below in the icon description area!

If you visit the Replication Area you will be able to get access to the API Logfile, the Replication Logfile and the DNS Server Manager. You can get informations how to set up a DNS Server Connection below. The API Logfile Serves as Log for Requests to and from this server to other dns servers. This log may not be fully reliable, cos not all areas are covered, to prevent useless database storage beeing used by this logs. In some cases, if there are errors, you may get more insights in that area about this area. Entries are marked with [in] or [out]. [in] describes an incoming request, to this server. [out] defines an outgoing request to another server.

In the Replication Logfile you can see the complete output of the replication process (made by the cronjob sync.php) and get deep insights at the operations which have been executed. The most of this information is covered in the UI and you normally do not need to look inside that files. But it can defenitly help if you are searching errors.

Besides that in the Domain section, you are able to see domains which have been replicated, or fetched localy. In the Master Domains section you will see Domains which have been locally fetched. This domain will always be threated as master domains. You shall not have other slave domains registered locally, this software predicts that you do only have master domains set up in the local servers named.conf.local and named.conf.default-zones (or the file you set up in settings.php). This is okay for as much as all configurations, if you are using this software to replicate entries. The fetched local Master Domains, which will may be replicated to other servers, can be all seen with the actual status in the Domain->Master Domains Section. More informations about the icons below in the documentation.

In the Domains section you can click on the button "Slave Domains". You will than see all domains which have been fetched from other DNS Servers, which are markes as "Master Server" in the DNS Server Management. More informations about the status icons below in the documentation.

You will have a conflict, if a domain exists on the local server as master domain and on a slave server as well, or is 2 slave servers deliver the same master domain. In this case the Conflict will be shown in the "Conflicts" Section. You have the possibility to prefer a domain in the Domain->Master Domains or Domain->Slave Domains section. This domain will than be preferred if there are any conflicts. If you do not solve a conflict, this domain wont be registered to the local bind instance, so it is always neccessary to solve conflicts if the domain should be active. The conflicts section ist just for conflicts overview and theire solutions and which servers are affected (where the conflicst occurs from). You can solve the Conflicts by prefering a domain. unsolved Conflicts are marked yellow and solved conflicts are marked as green.

By default the local Master domains are preferred, to prevent conflicts with slave servers from de-registering domains!

If you are in the Domains Section Domains Overview, you can view the zonedata if the domain which has been fetched external at the replication or locally. If there are errors in the zonedata, which is shown by an icon - you can get the output of the named-zonecheck command in the "Details" button window. There you will see any additional informations about this domain and errors that may occured durring the sync.php execution (which handles all stuff which is done in the background). You can disable Replication on Local Master Domains, this domains than will not be replicated to other servers, which are asking for replication.

Keep in mind that the Domain Section only gets updated, when the cronjob sync.php is running. At the top of the page you can see (in green) the date and time where the data has last been updated. If you want to update that data you need to run sync.php manually or wait until your cronjob cycle runs it. (See Installation)

Types of Server Connections

In the case you have a System with Master Domains and Slave Servers only dedicated for this case, you would install this software on both servers. On the master server you would setup the Slave Server marked as Slave Server, with related Website URL Where this panel is running, an token which is identical to the one on the other server and the servers ip adr. Is is important that the IP Adr. of the external server is valid! Otherwhise the replication Process wont work! You need at least one valid ipv4 or ipv6 where the other nameserver is reachable.
On the Slave Server now, you would install that panel as well and set up the master server there in the Replication->Mange DNS Server Area. You can see Status Icons for Server if the connection is possible which are explained below in the documentation.

Besides that it is possible to use "hybrid" connections. That means that every server running with this panel can act as slave and master server at the same time. The conflict section comes very in handy here.

Informations about Domain Prefering: If you Prefer a Domain on Server 1, it will not be preferred on Server 2. So you may have to prefer a domain on several servers ( depend on the number of your servers ) to solve all conflicts which exist.

Server Tokens

If you have at Server 1 a Token "123" for example, and on Server 2 the same Token configured "123" on the other server, than the replication process and all other API Requests will work. You need the exact same token for server connections on both servers existing. Keep sure that the Token you want to use to replicate, exists on both servers.

Disable Replication for Master/User Domains

You can disable "to-slave" replication for single master domains in the webinterface.

Add a Slave DNS Server

Look at the Section "Types of Server Connections" for informations about how to add servers.

Zone Checks

Every zone file which will be used, will checked with the command named-checkzone before it gets written for the configuration. If that command fails, the last working file gets in affect. If there is no working file the domain will not be active. You will get notification icons on that domain for every case. Look at the status icon information below in the documentation to get informations about this status icons. If a Zone is Using its fallback zonefile, or failing verifying the new file, you can get the error string of "named-checkzone" in the domains details section.

MySQL Debugging for Developers

If you are a developer or you want to change the software you can activate the mysql_debugging mode in the settings.php. This will let the "debug" section appear where you can see mysql errors, if there are some. This section is only for interest for developers. As now this software has been fully tested and this section is pretty much useles for normal users of this software. You can not see any informations about replication or other stuff here. Just MySQL Errors. Do not panic if there are some. The most of these errors occur if the table structure gets regenerated, but they will fix themselves.

Caution to not lose your login password!

Be careful to remember your password, there is no easy way to restore it - if you lose it...
If you lose your administrator password and you have no access to a user with permission for user mangement, than you have to reset the users table. Do this by going into your mysql server and delete the dnshttp_user table. If you visit the page after that, the table will be restored with the initial login information you can see below!


In case you dont want to delete your users table:

If you lose your admin password and you have no other administration account to access the user management, than the only thing you can do is to change the password manually in the database. Look at the users table and replace the string in the password field with a new bcrypt php string with your password. You can find different generators in the internet where you can generate this strings.

If you delete the table, all users will be deleted, but the password and administrative user will be reseted to the default login you can see at the end of the readme. This is an easier way, if you dont have to keep the users table.

Inside the webinterface is no password restore function and is not planned, but you can change account passwords if you are able to login.

Server Status Icon Explanation

Status Icon Informations:
The status of a server is updated if the cronjob sync.php does execute. Otherwhise you can update the status icons by pressing "recheck" on the server you want to check. It may be good to take a look at this values from time to time, but more important is to take a look at the replications log. All necessarry informations you need to know about the replication are written there. (The Replication Protocol Section unter the tab Replication).

API Status Icons:
API-Ping: Ok
- API is connectable and OK for Replication
API-Ping: Fail
- API is not connectable (API Path/URL is Wrong or other instance has errors at the remote server)
API-Ping: Unchecked
- API connectivity has not been checked yet (will be by cronjob or with button "recheck")

IP-Ban Status Icons:
IP-Ban: Banned
- This server is banned in the others server ip blocklist. You need to remove it on the other servers interface or wait until a remote servers daily cronjob may remove this ban... (May caused by to many tries with wrong tokens!)
IP-Ban: Unchecked
- IP Ban State has not been checked yet (will be by cronjob or with button "recheck")
IP-Ban: Ok
- Server is okay for replication, as this server is not banned remotely...

Token Status Icons:
Token-Status: Invalid
- The configured token for the server does not match the token which is on the remote server for this connection... (They need to match, please check on remote site or change this servers key to the same as on the remote site! This error may also occur, is there is another API Error.
Token-Status: Unchecked
- Token State has not been checked yet (will be by cronjob or with button "recheck")
Token-Status: Ok
- Server is okay for replication, as this server has a valid token handshake...

Domain Status Icons:
Domains: [NUMBER]
- Number of the Domains existand on the external server!
Domains: Unchecked
- Still not checked for Domains, will be done on cron-run or if you press "recheck"!

Domain Status Icon Explanation

Status Icon Informations:
The status of a server is updated if the cronjob sync.php does execute. It may be good to take a look at this values from time to time, but more important is to take a look at the replications log. All necessarry informations you need to know about the replication are written there. (The Replication Protocol Section unter the tab Replication).

Domain Status Icons:
Active
If label above is shown, the Domain is active and registered with bind!
Inactive
If lavel above is dhown, the Domain is not registered with bind and not active, this can be due to an error and maybe should been checked.. ;_; (But it is just saying, that this domain is not used in bind currently, and not able to query at the server. If you have no problem with that, leave it like that is.
Waiting for Cron
If you see lavel above, the domain is waiting for the next cronjob run to update its status!
Disabled
If you see the label above, this domain had been in a conflict which has been solved, by choosing another domain as preferred!

Conflict Icons:
Conflicted
If label above is shown on domain, it is inactive, because if has a conflict (duplicated domain on another slave server or at this master server). This domain will not be registered to the local system, until the conflict has been solved by preferring a domain via this interface.
Solved Conflict
If label above is shown, the domain has a conflict and is duplicated on this servers domain table. But this conflict has been solved by choosing a prefered domain! So all is alright and no steps are required, you can see more information about conflicts and if they are solved in the "Conflicts" Area...

Domain Info Icons:
Preferred
If label above is shown on domain, this domain is preferred and will be used if there are any conflicts for this specific domain!
Replication Disabled
If you see label above on a domain, this domain has been set to "not replicate". This master domain on this local server will than not be replicated to other remote servers, if they are asking for the locals server domains list...

Zonedata Status Icons:
New Zonedata invalid
Using Failsafe...
If this label above is shown, there is an error in the domains content data and it has not been registered with bind.
Zonedata Error
If label above is shown, the domain has a zonedata error on the new domains zonedata, the old (functioning) zonefile will than be used, until the new zonedata is valid...
